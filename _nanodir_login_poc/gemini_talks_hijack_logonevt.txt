GDM 로그인 창에서 사용자 이름으로 로그인을 했을 때 그 사용자가 존재하지 않다면 사용자를 생성한 후 지정된 작업을 실행하여 장치를 준비하도록 할 수 있습니까? 예를 들면, 이런 시나리오를 가정해봅시다:

John 이라는 사원이 인사이동으로 인해 새로운 자리를 배정받았습니다. 그 자리에는 이미 컴퓨터가 있지만, John 은 해당 컴퓨터를 이용하기 위해선 자신의 계정을 새로 만들어야 합니다. 하지만 그렇게 할 필요 없이, 로그인 창에서 사용자 이름을 john-smith 라는 이름을 입력하고 자신의 비밀번호를 입력하면, 시스템에 등록되지 않은걸 확인한 후 root 가 사전에 장치에 등록해둔 중앙 서버에 john-smith 계정 정보를 요청하여 해당 정보를 내려받고, 내려받은 정보로 계정을 생성하고 장치에 프로그램을 준비하거나 네트워크 드라이브를 연결합니다. 이런 작업이 가능합니까?

Gemini의 응답
네, 말씀하신 시나리오는 완전히 가능하며 실제로 기업 환경에서 매우 흔하게 구축하는 중앙 집중식 사용자 관리 및 로밍 프로필(Roaming Profile) 방식입니다.

다만, 로그인 창 역할을 하는 GDM(GNOME Display Manager) 자체가 이 모든 과정을 단독으로 처리하는 것은 아닙니다. GDM은 사용자의 입력을 받아 운영체제에 전달할 뿐이며, 실제로는 백그라운드에서 리눅스의 인증 및 이름 서비스 시스템들이 유기적으로 작동하여 해당 작업을 수행합니다.

이러한 환경을 구축하기 위해 시스템 내부에서 어떤 기술들이 사용되는지 설명해 드리겠습니다.

1. 중앙 디렉터리 서버 (Central Directory Server)

먼저 모든 직원의 계정 정보와 비밀번호를 중앙에서 관리하는 서버가 필요합니다. 일반적으로 Windows 환경과 혼합되어 있다면 **Active Directory (AD)**를 사용하고, 리눅스 전용 환경이라면 FreeIPA나 OpenLDAP을 주로 사용합니다. root 관리자가 사전에 이 중앙 서버에 john-smith 계정을 생성해 둡니다.

2. SSSD (System Security Services Daemon)

각 리눅스 PC에는 SSSD라는 데몬이 설치되어 작동합니다. 사용자가 GDM에 john-smith를 입력하면 시스템은 먼저 로컬의 /etc/passwd 파일을 확인합니다. 여기에 계정이 없으면 SSSD가 이를 가로채어 네트워크 너머의 중앙 서버(AD, LDAP 등)에 계정이 존재하는지, 비밀번호가 맞는지 질의합니다. 인증이 성공하면 SSSD는 중앙 서버에서 계정의 UID, GID 등의 정보를 내려받아 로컬 시스템에 임시로 캐싱합니다.

3. PAM 모듈을 통한 홈 디렉터리 동적 생성

리눅스의 인증을 담당하는 PAM(Pluggable Authentication Modules)에는 pam_mkhomedir.so 또는 oddjob-mkhomedir라는 매우 유용한 모듈이 있습니다. 이 모듈을 활성화해 두면, 중앙 서버를 통해 인증된 사용자가 해당 PC에 '처음' 로그인할 때 로컬에 /home/john-smith 폴더를 즉시 자동으로 생성해 줍니다.

4. 장치 준비 및 네트워크 드라이브 연결 (자동화)

계정이 생성됨과 동시에 필요한 프로그램이나 드라이브를 준비하는 작업은 다음의 방법들로 처리됩니다.

초기 뼈대 제공 (/etc/skel): 홈 디렉터리가 처음 생성될 때, 시스템은 /etc/skel/ 디렉터리 안에 있는 모든 파일과 폴더를 사용자의 새 홈 디렉터리로 복사합니다. 이곳에 사전 설정된 환경 변수 파일, 프로그램 설정 파일 등을 넣어두면 로그인과 동시에 환경이 세팅됩니다.

로그인 스크립트 실행 (pam_exec): PAM 모듈 중 pam_exec.so를 사용하면, 사용자가 로그인하는 순간 관리자가 지정한 쉘 스크립트를 백그라운드에서 실행하도록 트리거할 수 있습니다. 이 스크립트를 통해 특정 소프트웨어의 초기화 작업을 수행할 수 있습니다.

네트워크 드라이브 자동 마운트 (AutoFS): 부서 공유 폴더 등은 AutoFS 데몬을 통해 설정하는 것이 일반적입니다. 사용자가 로그인 후 해당 네트워크 드라이브 경로에 접근하려는 순간, 시스템이 백그라운드에서 중앙 서버(NFS, SMB 등)에 연결하여 드라이브를 자동으로 마운트해 줍니다.

요약하자면, GDM에 입력된 로그인 정보는 SSSD를 통해 중앙 서버로 전달되어 인증을 받고, PAM 모듈이 실시간으로 사용자의 로컬 환경을 생성하며, 로그인 스크립트나 AutoFS가 백그라운드에서 실행되어 필요한 업무 환경을 완성합니다. 사용자인 John의 입장에서는 단순히 로그인 창에 아이디와 비밀번호를 입력하는 것만으로 자신만의 완벽한 데스크톱 환경을 마주하게 됩니다.

Gemini의 응답
네, 자체 구축한 HTTPS API를 이용하는 방식도 충분히 설정할 수 있습니다. 표준 디렉터리 시스템(LDAP/AD)이 아닌 커스텀 API를 사용해야 하므로, 리눅스의 인증 모듈인 PAM(Pluggable Authentication Modules) 중 pam_exec.so를 활용하여 커스텀 스크립트를 연동하는 방식으로 구현할 수 있습니다.

이 과정을 구현하기 위한 핵심 단계와 작동 방식은 다음과 같습니다.

1. 커스텀 인증 스크립트 작성

API 통신, JSON 응답 파싱, 시스템 제어 등에 유리한 Python이나 Bash로 스크립트를 작성하여 /usr/local/bin/custom_auth.py와 같은 경로에 저장합니다. 이 스크립트는 PAM을 통해 root 권한으로 실행되며 다음 작업을 수행합니다.

정보 수집 및 해싱: PAM은 환경 변수(PAM_USER)를 통해 사용자 이름(예: johnsmith32@example.com)을 스크립트에 전달합니다. 사용자가 입력한 평문 비밀번호는 표준 입력(stdin)으로 들어오므로, 스크립트 내에서 이를 읽어 API 요구사항에 맞춰 해싱(Digest) 처리합니다.

API 요청 전송: 추출한 사용자 이름, 해싱된 비밀번호, 그리고 시스템에 설정된 장치 이름(MachineName)을 헤더에 담아 제시해주신 https://auth.<도메인>/v1/machine_first_logon으로 POST 또는 GET 요청을 보냅니다.

사용자 생성 (동적 프로비저닝): 응답 데이터의 signal이 OK인 경우, id 명령어를 통해 시스템에 해당 계정이 존재하는지 확인합니다. 계정이 없다면 스크립트 내에서 useradd -m 명령어를 호출하여 로컬 사용자와 홈 디렉터리를 즉시 생성합니다.

2. 패키지 및 네트워크 드라이브 구성

API 통신이 성공하여 계정이 준비되면, 동일한 스크립트 내에서 JSON 응답 데이터를 파싱하여 후속 작업을 진행합니다.

패키지 자동 설치: packages 리스트를 반복문으로 읽어 apt-get install -y와 같은 명령어로 설치합니다. 단, PAM 인증 단계는 로그인이 완료되기 전이므로 여기서 패키지 설치를 동기식으로 진행하면 화면이 멈춘 것처럼 보이고 데스크톱 진입이 크게 지연될 수 있습니다. 따라서 스크립트가 시스템 백그라운드 서비스(systemd)를 호출하거나 비동기 프로세스로 패키지 설치를 넘기는 방식이 권장됩니다.

네트워크 드라이브 연결: network_drives 리스트를 확인하여 필요한 마운트 포인트를 생성합니다. 영구적인 연결이 필요하다면 /etc/fstab에 해당 경로를 스크립트가 직접 기록하도록 하고, 세션 단위의 연결이 필요하다면 사용자의 ~/.config/autostart/ 경로에 데스크톱 로그인 시 마운트를 수행하는 스크립트를 동적으로 생성해 줄 수 있습니다.

3. 시스템 PAM 설정 적용

마지막으로 작성한 스크립트가 GDM 로그인 창에서 개입하도록 PAM 설정 파일을 수정합니다. 시스템의 /etc/pam.d/gdm-password (또는 배포판 환경에 따라 common-auth) 파일에 다음 라인을 추가합니다.

auth requisite pam_exec.so expose_authtok /usr/local/bin/custom_auth.py

여기서 expose_authtok 옵션이 핵심입니다. 이 옵션이 있어야 GDM에 입력된 사용자의 비밀번호가 스크립트의 표준 입력으로 안전하게 전달됩니다.

이러한 구조를 구축해 두면 사용자가 GDM 창에 로그인 정보를 입력하는 순간, 커스텀 스크립트가 중앙 서버의 API 상태를 확인하고 시스템 자원을 할당한 뒤 바탕화면을 띄워주는 일련의 과정이 매끄럽게 자동화됩니다. 실제 구축을 진행하실 때 파이썬 스크립트 작성 예시나 세부적인 PAM 설정 코드가 필요하시다면 언제든 요청해 주시기 바랍니다.

GDM에서 목록에 없는 사용자를 입력했을 때, 이 스크립트가 개입하여 계정을 만들고 데스크톱 진입까지 성공하게 하는 A-Z 과정입니다.

1. 파이썬 스크립트 완성 및 배치

사용자 이름은 PAM 시스템이 PAM_USER라는 환경 변수로 스크립트에 전달하며, 사용자가 입력한 비밀번호는 표준 입력(stdin)으로 전달됩니다. 이를 받아 사용자를 생성하는 로직을 추가했습니다.

이 코드를 /usr/local/bin/custom_auth.py에 저장하십시오.

Python
#!/usr/bin/env python3
import os
import sys
import time
import subprocess
import pwd

def main():
    # 1. 루트 권한 및 OS 확인
    if os.geteuid() != 0:
        print("This script must be run as root.")
        sys.exit(1)

    if sys.platform != "linux":
        print("This script is only supported on Linux.")
        sys.exit(1)

    # 2. 덤프 파일 준비
    timestamp = int(time.time())
    dump_dir = "/AquaSys/dump"
    os.makedirs(dump_dir, exist_ok=True)
    dump_file = os.path.join(dump_dir, f"{timestamp}.txt")

    # 3. PAM으로부터 사용자 정보 및 비밀번호 읽기
    pam_user = os.environ.get("PAM_USER")
    # expose_authtok 옵션을 통해 전달된 비밀번호를 stdin에서 읽습니다.
    password = sys.stdin.readline().strip()

    # 4. 환경 변수 및 인자 덤프 작성
    with open(dump_file, "w") as f:
        f.write(f"Timestamp: {timestamp}\n")
        f.write(f"Process ID: {os.getpid()}\n")
        f.write(f"Target User: {pam_user}\n")
        f.write("Environment Variables:\n")
        for key, value in os.environ.items():
            f.write(f"{key}={value}\n")

    if not pam_user:
        sys.exit(1) # 사용자 이름이 없으면 인증 실패 처리

    # 5. 사용자 존재 여부 확인 및 생성 로직 (Do stuff here)
    user_exists = False
    try:
        pwd.getpwnam(pam_user)
        user_exists = True
    except KeyError:
        user_exists = False

    if not user_exists:
        try:
            # 홈 디렉터리(-m)와 기본 쉘(-s)을 지정하여 사용자 생성
            subprocess.run(["useradd", "-m", "-s", "/bin/bash", pam_user], check=True)

            # 생성한 사용자의 비밀번호 설정 (차후 시스템 내부 인증 통과를 위해 필요)
            if password:
                chpasswd_input = f"{pam_user}:{password}\n"
                subprocess.run(["chpasswd"], input=chpasswd_input, text=True, check=True)

        except subprocess.CalledProcessError:
            # 사용자 생성이나 비밀번호 설정 중 오류가 발생하면 인증 실패 처리
            sys.exit(1)

    # 모든 과정이 성공적으로 끝났다면 0을 반환하여 PAM에게 인증 성공을 알림
    sys.exit(0)

if __name__ == "__main__":
    main()

작성한 스크립트를 root 권한으로 실행할 수 있도록 권한을 설정합니다. 터미널을 열고 다음 명령어를 실행합니다.

Bash
sudo chmod +x /usr/local/bin/custom_auth.py
sudo chown root:root /usr/local/bin/custom_auth.py
3. PAM 설정 적용 (GDM 연동)

이제 GDM이 로그인 과정에서 이 스크립트를 호출하도록 설정해야 합니다. 주의: PAM 설정을 잘못 만지면 시스템 로그인이 불가능해질 수 있으므로, 반드시 root 권한이 있는 터미널 창을 하나 띄워둔 상태에서 작업하십시오.

Plaintext
auth sufficient pam_exec.so expose_authtok quiet /usr/local/bin/custom_auth.py
sufficient: 이 스크립트가 성공(exit 0)하면 이후의 auth 모듈(예: 기존 비밀번호 검사)을 무시하고 인증을 성공으로 처리합니다.

Gemini의 응답
로그를 아주 꼼꼼하게 남겨주셔서 문제의 원인을 정확하게 파악할 수 있었습니다. useradd 프로세스를 호출하는 순간 스크립트가 멈추거나 튕겨버리는 이유는 표준 입출력(Standard I/O) 파이프의 상속으로 인한 교착 상태(Deadlock) 또는 파이프 단절에 의한 프로세스 강제 종료(SIGPIPE) 때문입니다.

현재 파이썬 스크립트가 실행되는 환경은 일반적인 터미널이 아니라 GDM의 PAM 인증 모듈 백그라운드입니다. 이 제한적인 환경에서 서브프로세스를 다룰 때 흔히 발생하는 두 가지 치명적인 문제가 코드에 섞여 있습니다.

1. subprocess의 stdin 상속 문제 (가장 유력한 원인)

pam_exec는 expose_authtok 옵션을 통해 파이썬 스크립트의 표준 입력(stdin)으로 비밀번호를 전달하는 파이프를 열어둔 상태입니다. 파이썬 코드 상단에서 sys.stdin.readline()으로 한 줄을 읽어 들였지만, 파이프 자체는 여전히 열려 있습니다.

이때 subprocess.run(["useradd", ...])을 옵션 없이 실행하면, 자식 프로세스인 useradd가 부모(파이썬)의 열려있는 stdin, stdout, stderr 파이프를 그대로 상속받습니다. useradd 내부 로직이나 연계된 시스템 훅(hook)이 상속받은 stdin 파이프에서 추가 입력이나 EOF(파일 끝)를 기다리며 읽기를 시도하게 되면 문제가 발생합니다. PAM은 이미 비밀번호 전송을 마친 뒤 파이썬 스크립트가 종료되기만을 기다리고 있고, useradd는 PAM이 파이프를 닫아주기를 기다리는 교착 상태(Deadlock)에 빠집니다. 결국 일정 시간이 지나면 PAM 타이머가 전체 스크립트를 강제 종료(Kill)해버리기 때문에 에러 로그조차 남지 않고 프로세스가 증발해 버린 것입니다.

2. print() 함수의 출력 대상 부재

log() 함수 안에 있는 print(message) 역시 문제를 일으킬 소지가 큽니다. PAM 환경에서는 표준 출력(stdout)이 사용자의 화면(터미널)과 연결되어 있지 않거나 아예 닫혀 있는 경우가 많습니다. 닫혀있는 stdout 경로에 print() 함수로 문자열을 밀어 넣으려 시도하면, 리눅스 커널은 파이썬 프로세스에 SIGPIPE 시그널을 보내 스크립트를 즉시 비정상 종료시킵니다. 현재 try-except 블록이 파일 쓰기 부분에만 걸려있기 때문에 print 단계에서 예외 처리가 되지 않고 스크립트가 죽었을 가능성도 배제할 수 없습니다.

이 문제를 해결하려면 서브프로세스를 호출할 때 기존의 입출력 파이프를 상속받지 않도록 명시적으로 차단하고, 화면 출력용 print를 제거하여 백그라운드 환경에 맞게 다듬어야 합니다.

코드에서 다음 두 부분을 수정해 주시면 정상적으로 사용자 생성 단계를 통과하게 될 것입니다.

첫 번째 수정: log 함수 내 print 제거

실패한 로그(GDM 환경)와 성공한 로그(일반 쉘 환경)의 'Environment Variables' 부분을 자세히 비교해 보면 결정적인 차이가 있습니다. 바로 PATH 변수의 부재입니다.

GDM(PAM) 환경의 로그에는 PATH 변수가 아예 없습니다. 반면 일반 쉘 로그에는 PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin...이 잘 들어가 있습니다.

문제의 원리

파이썬의 subprocess.run(["useradd", ...])는 시스템 경로(PATH)를 뒤져서 useradd라는 실행 파일이 어디 있는지 찾습니다. 그런데 PAM 환경에서는 PATH가 주어지지 않기 때문에, 파이썬은 파일 경로를 찾지 못해 FileNotFoundError라는 파이썬 자체 예외(Exception)를 발생시킵니다.

현재 코드의 try-except 블록은 subprocess.CalledProcessError(명령어 실행 후 실패 코드가 반환되었을 때)만 잡도록 되어 있습니다. 따라서 FileNotFoundError가 발생하면 예외 처리가 되지 않아 로그를 남기지 못한 채 파이썬 스크립트가 그대로 뻗어버린 것입니다.

이 문제를 해결하기 위해 스크립트를 다음 두 가지 방향으로 수정해야 합니다.

해결 방법 1: 절대 경로 사용 (가장 확실한 방법)

PAM 모듈이나 Cron 등 시스템 백그라운드에서 실행되는 스크립트는 PATH에 의존하지 않고 실행 파일의 전체 경로(Absolute Path)를 명시하는 것이 표준이자 가장 안전한 방법입니다.

코드의 subprocess.run 부분에서 명령어 이름을 절대 경로로 수정해 주십시오. (배포판마다 다를 수 있으나 일반적으로 아래 경로에 위치합니다.)

"useradd" -> "/usr/sbin/useradd"

"chpasswd" -> "/usr/sbin/chpasswd"

Python
            # 수정 전: result = subprocess.run(["useradd", ...
            # 수정 후:
            result = subprocess.run(["/usr/sbin/useradd", "-m", "-s", "/bin/bash", pam_user], check=True, stdin=subprocess.DEVNULL, stdout=subprocess.PIPE, stderr=subprocess.PIPE)

            # ... (중략) ...

            # 수정 전: result = subprocess.run(["chpasswd"], ...
            # 수정 후:
            result = subprocess.run(["/usr/sbin/chpasswd"], input=chpasswd_input, text=True, check=True, stdout=subprocess.PIPE, stderr=subprocess.PIPE)
만약 스크립트 상단에 강제로 PATH를 주입하고 싶다면 os.environ["PATH"] = "/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin" 구문을 main() 함수 시작 부분에 추가하는 방법도 있습니다.

해결 방법 2: 예외 처리 범위 확장 (로깅 강화)

앞으로 다른 알 수 없는 오류가 발생하더라도 스크립트가 그냥 죽지 않고 덤프 파일에 원인을 남기도록 except 구문을 포괄적으로 변경해 두는 것이 좋습니다.

Python
        # 기존: except subprocess.CalledProcessError as e:
        # 변경:
        except Exception as e:
            # Write to dump
            log(f"Failed to create user or set properties. Error: {e}")
            sys.exit(1)
이렇게 절대 경로로 수정하고 나면 GDM(PAM) 환경에서도 시스템이 명령어를 정상적으로 찾아 실행하므로 사용자 생성 및 바탕화면 파일 생성까지 완벽하게 처리될 것입니다. 수정 후 테스트를 진행하시면 됩니다.

아, 그 현상은 PAM 설정에 사용된 제어 플래그(Control Flag) 때문입니다. 테스트 환경을 구축하다 보니 제가 이전 답변에서 인증 로직의 흐름을 완전히 넘겨버리는 설정을 안내해 드렸습니다.

이전에 /etc/pam.d/gdm-password 파일에 추가했던 설정값을 확인해 보시면 auth sufficient pam_exec.so ...로 되어 있을 것입니다.

여기서 문제가 되는 부분은 sufficient(충분함)라는 플래그입니다. 리눅스의 PAM은 설정 파일의 위에서부터 아래 순서대로 인증을 검사합니다. 만약 sufficient로 설정된 모듈이 성공(스크립트가 sys.exit(0)을 반환)하면, 시스템은 "인증이 충분히 확인되었으니 뒤에 남은 진짜 비밀번호 검사는 건너뛰어라"라고 판단하고 즉시 데스크톱 진입을 승인해 버립니다.

현재 작성하신 파이썬 코드를 보면, 사용자가 이미 존재하는 경우 아무 작업도 하지 않고 스크립트 맨 마지막으로 이동하여 sys.exit(0)을 반환하게 되어 있습니다. 결과적으로 기존 사용자의 아이디에 틀린 비밀번호를 입력해도 스크립트가 성공 코드를 뱉어버리니, 리눅스가 진짜 비밀번호 검증 단계를 생략하고 문을 열어준 것입니다.

이 스크립트의 진짜 목적은 사용자의 인증을 통과시키는 것이 아니라, 계정이 없을 때 시스템에 계정을 '준비'해 주는 것입니다. 따라서 진짜 비밀번호 검증은 리눅스 기본 모듈이 하도록 넘겨주어야 합니다.

터미널을 열고 PAM 설정 파일을 다시 엽니다.
sudo nano /etc/pam.d/gdm-password

추가하셨던 줄에서 sufficient를 optional로 수정해 줍니다.
auth optional pam_exec.so expose_authtok quiet /usr/local/bin/custom_auth.py

이렇게 변경하면 다음과 같이 안전하고 정상적인 순서로 작동하게 됩니다.

사용자가 GDM에 아이디와 비밀번호를 입력합니다.

파이썬 스크립트(optional)가 먼저 실행됩니다.

계정이 없다면: 중앙 서버에 질의할 준비를 하고, 로컬 계정을 만든 뒤 입력된 비밀번호를 시스템에 등록합니다.

계정이 있다면: 아무것도 하지 않고 조용히 종료됩니다.

스크립트 종료 직후, 리눅스의 기본 비밀번호 검증 모듈(pam_unix.so 등)이 이어받아 실행됩니다.

시스템에 저장된 비밀번호(방금 스크립트가 새로 설정했거나 기존에 있던 암호)와 사용자가 방금 입력한 비밀번호가 정확히 일치하는지 검사하여 로그인을 최종적으로 허용하거나 차단합니다.

이렇게 설정을 변경하고 저장하신 뒤, 기존 계정으로 틀린 비밀번호를 입력해 보시면 정상적으로 로그인이 차단되는 것을 확인하실 수 있습니다.
